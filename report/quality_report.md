# Отчет о качестве данных

## 1. Введение

Данные представляют собой список, содержащий все транзакции, произошедшие в период с 01.12.2010 по 09.12.2011 для зарегистрированной в Великобритании онлайн-компании, не имеющей розничных магазинов. Многие клиенты компании являются оптовыми покупателями.

Заказчиком анализа может выступать интернет-магазин предоставляющий данные или магазин-конкурент.

С помощью этого датасета можно решить такие задачи интеллектуального анализа данных, как:
- Поиск ассоциативных правил;
- Сегментация клиентов;
- Описательная аналитика;

## 2. Паспорт датасета: 

В датасете 541909 строк, 8 столбцов  
Признаки:  
InvoiceNo       - Номер заказа  
StockCode       - Код товара  
Description     - Название/описание товара  
Quantity        - Количество товара в заказе (может быть отрицательным для возвратов)  
InvoiceDate     - Дата и время создания заказа  
UnitPrice       - Цена за единицу товара (в фунтах стерлингов)  
CustomerID      - Уникальный идентификатор клиента  
Country         - Страна доставки/клиента

## 3. Основные проблемы качества данных:  

| № | Тип проблемы | Описание | Потенциальный риск для анализа | Предлагаемое решение/ стратегия |
|---|--------------|----------|--------------------------------|---------------------------------|
| 1 | Пропущенные значения | В столбце `CustomerID` отсутствует ~25% значений. В `Description` — 0,27%. | Высокий риск для анализа, связанного с клиентами (сегментация, анализ поведения, RFM). | Анализировать клиентов только по записям с заполненным ID, с указанием этого ограничения в выводах. Не использовать CustomerID для задач, где он не критичен. Для `Description`: удалить строки или заполнить наиболее частым значением для данного `StockCode`. |
| 2 | Дубликаты и некорректные значения | 5 268 полностью дублирующихся строк. Отрицательные значения `Quantity` (10 624 записи) и `UnitPrice` (2 записи), указывающие на возвраты или корректировки. Нулевые `UnitPrice` (2 515 записей). | Дубликаты искусственно завышают метрики. Смешивание продаж и возвратов в одной выборке искажает закономерности. Товары с нулевой ценой требуют уточнения, иначе могут быть приняты за выбросы. | Удалить полные дубликаты. Разделить продажи и возвраты для раздельного анализа. Для `UnitPrice`: удалить или исследовать строки с отрицательной, нулевой ценой.|
| 3 | Выбросы в числовых признаках | Экстремальные значения в `Quantity` (мин: -80995, макс: 80995) и `UnitPrice` (макс: 38970). Метод IQR определяет >10% и >7% выбросов соответственно. | Выбросы искажают статистические показатели (среднее, стандартное отклонение). Возможно, являются ошибками ввода. | Исследовать природу выбросов. Выделить оптовые транзакции в отдельную категорию.|
| 4 | Проблемы качества категориальных данных | В `Description` много значений с начальными или конечными пробелами (1271 и 112 626 соответственно). Разное количество уникальных `StockCode` (4070) и `Description` (4223), что указывает на возможную несогласованность в описаниях товаров, их дублирование. | Пробелы приводят к созданию нескольких категорий для одного товара, что затрудняет анализ товарного ассортимента и поиск правил. | Очистить текст, привести его к общему виду. Сопоставить коды и описания, проверить случаи, когда одному `StockCode` соответствует несколько `Description` и наоборот. |
| 5 | Дисбаланс данных и контекст | Подавляющее большинство заказов (~90%) приходится на Великобританию. Данные за ограниченный период (1 год). Наличие оптовых покупателей смешивает паттерны B2C и B2B. | Модели, построенные на таких данных, не будут репрезентативны для других рынков. Сезонность и тренды за год могут быть неполными. Паттерны покупок оптовых и розничных клиентов отличаются, их смешение дает неясные результаты. | Отдельно анализировать заказы из Великобритании и за её пределами. Указать временные ограничения в выводах. Разделить клиентов на розничных и оптовых и анализировать эти группы отдельно. |

## 4. План улучшения качества:

**Высокий приоритет:**

- Удаление полных дубликатов.
- Корректировка текстовых полей.
- Разделение данных на продажи и возвраты.
- Удаление строк с UnitPrice <= 0.

**Средний приоритет:**
- Обработка пропусков CustomerID – удаление строк с пустым ID при клиенто-ориентированном анализе, сохранение при товарном анализе.
- Согласование StockCode и Description.
- Исследование выбросов в Quantity, UnitPrice.

**Низкий приоритет:**

- Отделение данных по Великобритании и другим странам; выделение возможных оптовых покупателей для раздельного анализа.
- Создание производных признаков – из InvoiceDate выделить месяц, день недели, час; рассчитать Revenue как Quantity * UnitPrice для транзакций продаж.
